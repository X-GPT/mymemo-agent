# Agent Workspace Implementation Plan

This document turns `docs/agent-workspace-technical-design.md` into an
implementation backlog for coding agents. It assumes there is no existing client
contract to preserve, so agents should implement the target vocabulary and
runtime model directly.

## Global Acceptance Criteria

- The public chat API uses `conversationId`, not client-facing `sessionId`.
- `runId` identifies one backend execution attempt and is propagated through
  chat-api, sandbox-daemon, gateway tokens, logs, and run events.
- Claude SDK resume state is named `agentSessionId` everywhere it is exposed or
  stored.
- The sandbox never receives provider keys, database credentials, or broad
  document credentials.
- LLM and document gateway routes enforce token audience.
- Document access exposed to the agent is one operation: `search_documents`.
- Local hydrated documents are treated as a working set, not as proof that the
  full remote corpus has been searched.
- Workspaces follow the target layout under
  `/workspace/conversations/{conversationId}/`.
- Durable workspace state is stored outside the sandbox through an explicit
  `WorkspaceStore` abstraction.

## Milestone 1: Core Runtime Identity

Goal: establish final vocabulary and IDs across chat-api, sandbox-daemon,
gateway, and tokens.

### Task 1: Replace Chat Request And Session Vocabulary

Update the chat API so the product-visible thread id is `conversationId`.
Remove client-facing `sessionId`. Internally, use `agentSessionId` only for
Claude SDK resume state. Generate a `runId` at request entry.

Acceptance criteria:

- `POST /api/v1/chat` rejects request bodies containing `sessionId`.
- `POST /api/v1/chat` accepts `conversationId` according to the chosen contract:
  either required from the caller or generated by chat-api when omitted.
- SSE emits the final client-facing event names:
  - `conversation_id`
  - `run_id`
  - `agent_session_id`
  - `sandbox_id`
  - `text_delta`
  - `done`
  - `error`
- No exported chat-api type uses `sessionId` to mean Claude SDK session state.
- Acceptance tests run locally:
  - schema unit tests directly exercise `ChatBodyRequest.safeParse(...)`
  - route-level integration tests call the real in-process Hono chat-api with
    `app.request(...)`
  - sandbox/orchestration is mocked; no E2B, gateway, database, local server, or
    provider API call is required

### Task 2: Add Run Identity Propagation

Pass `{ userId, conversationId, runId }` from the chat controller into sandbox
orchestration. Include those fields in the sandbox-daemon `/turn` request and
structured logs.

Acceptance criteria:

- Every sandbox turn has a stable `runId`.
- The daemon receives `conversation_id` and `run_id`.
- Logs include `userId`, `conversationId`, and `runId` on run-level events.
- Tests assert run and conversation IDs are propagated from chat route to the
  mocked sandbox turn request.

### Task 3: Add Token Audiences

Extend token claims with `aud: "llm" | "documents"`, `conversationId`, and
`runId`. Mint separate LLM and document tokens.

Acceptance criteria:

- LLM routes reject tokens with `aud: "documents"`.
- Document routes reject tokens with `aud: "llm"`.
- Token tests cover missing, invalid, and wrong audience values.
- Token tests cover expired tokens.
- The daemon receives separate `llm_token` and `doc_token` values.
- The sandbox environment never contains provider keys or database credentials.

## Milestone 2: Workspace Layout And Manifests

Goal: make the sandbox filesystem match the target workspace model.

### Task 4: Implement Sandbox Workspace Paths

Add a sandbox-daemon workspace module that creates and validates the target
workspace paths.

Target layout:

```text
/workspace/
  system/
  conversations/
    {conversationId}/
      docs/
      work/
      output/
```

Acceptance criteria:

- The daemon creates workspace directories before spawning the agent.
- The agent process runs from
  `/workspace/conversations/{conversationId}/work`.
- Path construction rejects or sanitizes malformed `conversationId` values that
  could escape the conversation workspace.
- Unit tests cover valid IDs, invalid IDs, and directory creation.

### Task 5: Implement Docs Manifest

Add `docs/manifest.json` support for hydrated documents.

Manifest entries should track:

- document id
- title
- local path
- source
- hydrated or updated timestamp
- run id that caused hydration

Acceptance criteria:

- A manifest is created when missing.
- Existing manifests are read and preserved.
- Malformed manifests fail closed with a clear error or are replaced only by an
  explicit recovery path covered by tests.
- Manifest writes avoid corrupt partial output.
- Unit tests cover empty, existing, and malformed manifests.

### Task 6: Add Durable Workspace Store Interface

Define a `WorkspaceStore` abstraction. Use a local filesystem adapter first if
the object storage provider is not finalized.

Required operations:

- `hydrateConversationWorkspace`
- `syncConversationWorkspace`
- `readDocsManifest`
- `writeDocsManifest`
- `appendRunEvent`

Durable path model:

```text
users/{userId}/conversations/{conversationId}/work/
users/{userId}/conversations/{conversationId}/output/
users/{userId}/conversations/{conversationId}/docs/manifest.json
users/{userId}/runs/{runId}/events.jsonl
```

Acceptance criteria:

- chat-api calls hydrate before a run.
- chat-api calls sync after a run succeeds, fails, or is canceled when a sandbox
  was leased.
- The local adapter follows the durable path model.
- Tests verify no cross-user, cross-conversation, or path traversal access is
  possible.

## Milestone 3: Document Search And Hydration

Goal: expose one document-facing operation to the agent.

### Task 7: Replace Agent-Facing Search/Fetch

Remove or hide agent-facing `mymemo-docs fetch`. Implement one operation named
`search_documents`.

`search_documents` should:

1. Search the remote document index through the unified gateway.
2. Rank or select top candidates within policy limits.
3. Fetch selected documents through the unified gateway.
4. Write fetched documents to local sandbox disk.
5. Update the docs manifest.
6. Return snippets plus local file paths.

Acceptance criteria:

- Agent-facing help shows one document operation.
- Search result includes `documentId`, `source`, `title`, `snippet`, and
  `localPath`.
- Repeated search returns `source: "already_local"` for documents already
  hydrated in the current conversation workspace.
- Tests cover no results, gateway errors, hydration success, and already-local
  documents.

### Task 8: Add Hydration Limits

Enforce document hydration limits in one policy module.

Acceptance criteria:

- Maximum documents per search is enforced.
- Maximum hydrated bytes per document is enforced.
- Maximum hydrated bytes per run is enforced.
- Limits are configurable through environment or a policy module.
- Oversized documents are not blindly written to disk.
- Tests cover each limit and the returned error shape.

### Task 9: Update Agent Prompt

Rewrite the agent prompt around the new document policy.

Acceptance criteria:

- The prompt says remote search is required by default for open document
  questions.
- The prompt says local documents are only the current working set.
- The prompt says local-only work is acceptable only when the user explicitly
  scopes the task to already-loaded, current, or just-found files.
- The prompt no longer instructs separate `search` then `fetch`.
- Prompt tests assert the remote-search policy is present.
- Prompt tests assert old `mymemo-docs fetch` wording is absent.

## Milestone 4: Run Lifecycle And Event Persistence

Goal: make runs auditable and replayable.

### Task 10: Add Run State Module

Create a run lifecycle module.

Required operations:

- `createRun`
- `appendRunEvent`
- `markRunCompleted`
- `markRunFailed`
- `markRunCanceled`

Acceptance criteria:

- Each run records start, sandbox lease, daemon start, agent events, hydration
  events, and completion or failure.
- Failed runs include a normalized error message.
- Run events are persisted as NDJSON.
- Tests cover successful, failed, and canceled lifecycles.

### Task 11: Wire SSE From Run Events

Map daemon events into run events. Map run events into client SSE events.

Acceptance criteria:

- Client-visible stream is derived from recorded run events.
- `done` is emitted only after successful completion.
- `error` is emitted on daemon failure, transport failure, or orchestration
  failure.
- Tests cover event ordering for success and failure.

### Task 12: Prepare Cancellation Contract

Add backend cancellation state and a sandbox/daemon cancellation hook.

Acceptance criteria:

- Run state model includes `canceled`.
- Sandbox or daemon abstraction exposes a cancellation method.
- Cancellation state transition is idempotent.
- Tests cover canceling active, completed, failed, and already-canceled runs.

## Milestone 5: Sandbox Leasing

Goal: move from per-turn sandboxes to warm conversation sandboxes.

### Task 13: Create Sandbox Lease Manager

Lease sandboxes by `{userId, conversationId}`.

Acceptance criteria:

- The same user and conversation reuse a healthy warm sandbox.
- Different users never share a sandbox.
- Different conversations do not share a workspace.
- Concurrent turns in the same conversation are rejected or serialized according
  to the selected policy.
- Tests cover lease reuse, isolation, and concurrent acquisition.

### Task 14: Add Idle Termination

Keep active sandboxes alive, retain idle sandboxes briefly, then sync and
terminate.

Initial policy:

```text
active run: keep sandbox alive
0-5 minutes idle: keep warm
>5 minutes idle: sync and terminate
```

Acceptance criteria:

- Fake-timer tests verify idle termination after the configured window.
- Active sandboxes are not terminated mid-run.
- Workspace sync is called before sandbox kill.
- Termination failures are logged and surfaced according to run policy.

### Task 15: Add Recycle Conditions

Recycle warm sandboxes when reuse is unsafe.

Acceptance criteria:

- Daemon health failure creates a new sandbox.
- System bundle version mismatch creates a new sandbox.
- Explicit risk signal creates a new sandbox.
- Tests cover health, version drift, and risk recycle paths.

## Milestone 6: Gateway Modularization And OpenRouter

Goal: keep one deployed gateway while separating responsibilities internally and
routing LLM traffic through OpenRouter by gateway policy.

### Task 16: Split Gateway Modules

Move gateway code into domain modules:

```text
apps/gateway/src/
  auth/
  llm/
  documents/
  db/
  server.ts
  env.ts
```

Acceptance criteria:

- Route behavior is unchanged after the split.
- Existing gateway tests still pass.
- Document routes are registered before the LLM catch-all.
- Token verification logic has one shared implementation.

### Task 17: Add OpenRouter Provider Adapter

Add OpenRouter as the default LLM upstream owned by the gateway.

Required environment:

- `LLM_PROVIDER`
- `OPENROUTER_API_KEY`
- `OPENROUTER_BASE_URL`
- `OPENROUTER_DEFAULT_MODEL`

Optional environment:

- `OPENROUTER_HTTP_REFERER`
- `OPENROUTER_APP_TITLE`
- model routing or fallback preferences

Acceptance criteria:

- No OpenRouter key reaches sandbox-daemon, the agent child process, or tool
  runtime.
- Gateway can stream OpenRouter responses through the existing sandbox-facing
  Claude SDK compatibility surface.
- Gateway injects OpenRouter credentials only on upstream requests.
- Provider errors are normalized and do not leak secrets.
- Tests cover upstream URL construction, request headers, and streaming response
  forwarding.

### Task 18: Add Model Policy

Centralize model selection and allowlist enforcement in the gateway.

Acceptance criteria:

- Gateway chooses a default model from configuration.
- Disallowed model requests are rejected or rewritten by policy.
- Usage attribution includes `userId`, `conversationId`, and `runId`.
- Tests cover default model selection, disallowed models, and attribution
  fields.

## Milestone 7: Hardening And End-To-End Verification

Goal: prove the full target architecture works safely.

### Task 19: End-To-End Happy Path

Add an end-to-end smoke path for the target runtime.

Acceptance criteria:

- A chat request creates a run.
- A sandbox is leased.
- The workspace is hydrated.
- The agent searches documents.
- Documents are hydrated locally.
- An answer streams back.
- The workspace syncs after the run.
- The run completes with persisted events.
- The hydrated docs manifest contains fetched documents.

### Task 20: Security Regression Tests

Add regression coverage for trust-boundary failures.

Acceptance criteria:

- Wrong-audience tokens fail closed.
- Expired tokens fail closed.
- Document scope widening attempts fail closed.
- Path traversal in `conversationId`, `userId`, or document ids fails closed.
- Tests assert provider keys and database credentials are not passed to the
  sandbox agent process.

### Task 21: Operational Docs

Update repository docs for the new runtime.

Acceptance criteria:

- Docs describe the new chat API request and SSE response shape.
- Docs no longer describe `sessionId` as client-facing.
- Docs no longer describe agent-facing `fetch_document` or `mymemo-docs fetch`
  as the intended workflow.
- Docs describe workspace lifecycle, run lifecycle, gateway provider config,
  token audiences, and sandbox idle policy.
