MyMemo agent workspace architecture
E2B runtime · unified gateway · VPC DB

Use E2B for execution. Keep documents behind the gateway.

The target design is a warm-on-demand AI workspace: the user gets durable continuity, the sandbox only runs while useful, and document access remains mediated by a VPC-capable unified gateway instead of direct database credentials.

Architecture Decision

Choose E2B now, not Fargate. E2B is better aligned with Bash-capable, prompt-influenced agent execution. AWS remains the durable data/control plane: database, unified gateway, object storage, audit, and state.
Runtime Warm-on-demand sandbox One logical workspace per user, but physical sandboxes are created or reused only around active work and killed after a short idle window.
Boundary No DB access from sandbox The sandbox never receives DATABASE_URL, AWS credentials, or broad document credentials. It asks a gateway for documents.

Private database reachable only from the unified gateway security group, never from E2B sandboxes.

The LLM module injects provider keys, enforces model/budget policy, and keeps LLM credentials outside the sandbox.

Sandbox Lifecycle

1. Request

User sends a message for a conversation or task.

2. Lease

Reuse a warm E2B sandbox for the user or create a new one.

3. Hydrate

Restore conversation workdir, selected artifacts, and likely docs.

4. Run

Agent works locally and can request more documents on demand.

5. Sync

Persist changed files, emitted events, fetched-doc manifest, and outputs.

6. Recycle

After 3-5 minutes idle, version drift, or risk signal, terminate sandbox.

Document Access Model

Open question

For broad document questions, remote search is the default even when local files seem relevant.

One tool

Agent calls search_documents(query). There is no separate agent-facing fetch step.

Internal hydrate

Gateway/daemon searches, ranks, fetches selected docs, and writes them to local disk.

Combined results

Tool returns already-local docs plus newly hydrated remote docs as local file paths.

Local process

Agent reads returned paths under the conversation docs directory and works locally.

Explicit local only

Skip remote search only when the user clearly says to use current or already-loaded files.

Sandbox Rules

  • Sandbox is a hydrated working copy, not the source of truth.
  • Conversation files live under /workspace/conversations/{conversationId}.
  • Local docs are a working set, not proof the whole corpus has been searched.
  • Documents are projected per conversation/run, not copied as a full user corpus.
  • Daemon/agent updates are handled by recycling idle sandboxes or starting new ones with the latest bundle.

Unified Gateway Rules

  • Use separate doc_token and llm_token.
  • Add and enforce aud, userId, conversationId, runId, scope, and expiry.
  • DB security group allows inbound only from the unified gateway.
  • search_documents internally searches, fetches, hydrates, dedupes, limits, and audits.

Cost Controls

  • Keep sandboxes warm only briefly after active work.
  • Persist state to object storage/Postgres before termination.
  • Hydrate only needed files to reduce startup and storage cost.
  • Use Fargate later only if AWS-native control outweighs sandbox-platform engineering cost.

E2B vs Fargate

Choice
Strength
Recommendation
E2B
Purpose-built agent sandbox runtime with the current daemon/orchestration model already in place.
Use now. Add warm-on-demand leasing, durable workspace sync, and unified-gateway-backed document access.
Fargate
AWS-native task runtime with strong VPC/IAM integration, but it is a generic container platform.
Consider later if you want to own the sandbox platform inside AWS and need tighter enterprise controls.
Always-on per-user runtime
Best continuity and lowest follow-up latency, but idle cost scales directly with users.
Avoid as the default. Use short warm windows and recycle aggressively.

Target architecture: E2B runs the untrusted agent only while useful; the AWS-hosted unified gateway and stores own data authority, durable state, audit, provider-key injection, and VPC-only database access. The sandbox can discover missing documents through controlled on-demand retrieval, but it never connects to the database directly.
