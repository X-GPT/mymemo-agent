AS-BUILT ARCHITECTURE · X-GPT/chat-api
PR #116 · scope-bound document-gateway ↗
What this PR builds

The agent holds a token,
never the key.

Every real credential moves out of the E2B sandbox into a single dedicated control plane — one gateway that serves both the model and the user's documents. chat-api mints one short-lived, scope-bound bearer token per turn; the gateway is the only place the real ANTHROPIC_API_KEY and DATABASE_URL live — now co-located in one process.

01

Topology & trust boundary

Two zones. Credentials are color-coded: ● provider key, ● KB database URL, ● bearer token.

Trusted infrastructure · your servers
chat-apimints token
Orchestrator. Validates identity, opens the SSE stream, creates a fresh E2B sandbox per turn (torn down in a finally), and signs a per-turn, scope-bound llm_token. No provider key.
gatewayANTHROPIC_API_KEY DATABASE_URL
Stateless service, two jobs behind one token-verify. LLM proxy: injects the real x-api-key and proxies a strict path allowlist (/v1/messages, /v1/messages/count_tokens) to Anthropic. Document reader: on /v1/documents/* it enforces the signed scope (global / collection / document) server-side, pins every call to the token's userId (the KB workspace_id), and reads the MyMemo knowledge base (Postgres) directly — FTS over passages. The document routes are registered before the catch-all Anthropic proxy. The only holder of ANTHROPIC_API_KEY and the read-only DATABASE_URL — both now live in this one process.
E2B sandbox · untrusted
sandbox-daemonno secrets
Long-running in-sandbox server. Per turn it just spawns the agent, forwarding the token + gateway URLs into its env. No provider key, no document key, no database.
agent · claudebearer token only
srt-isolated (bwrap on Linux · sandbox-exec on macOS), network egress allowlisted to the gateway hosts only. Env: ANTHROPIC_BASE_URL/ANTHROPIC_AUTH_TOKEN + MYMEMO_DOC_GATEWAY_URL/MYMEMO_DOC_TOKEN. Reaches documents on demand via the mymemo-docs CLI — nothing worth stealing, nowhere to send it.
trust boundary
The agent has nowhere else to reach — srt's network allowlist denies every egress except the single gateway host. Its only token is a 10-minute, single-user Bearer token whose scope is signed in, so a prompt-injected agent can't widen its document access, has nothing worth stealing, and nowhere to send a stolen secret.
EXTERNAL  ·  api.anthropic.com — reached only by the gateway (with the injected x-api-key)  ·  MyMemo KB Postgres (RDS) — read only by the gateway (with DATABASE_URL)

● the real keys — co-located

Both now live on the one gateway. Each is still used in one place and crosses one external edge — but a single process holds both, a wider blast radius than two services.

gateway  set x-api-key  →  api.anthropic.com
gateway  DATABASE_URL  →  MyMemo KB Postgres

● the token — travels

Minted with scope, carried in the turn body, set as agent env, presented as Bearer to the gateway (both route families), verified, discarded.

chat-api → daemon → agent → gateway (/v1/messages · /v1/documents/*) ✓
02

One turn, end to end

From POST /api/v1/chat to a streamed answer.

Client → chat-api
POST /api/v1/chat with identity headers; chat-api opens the SSE response.
Create the sandbox
createSandbox — a fresh E2B sandbox per turn (no reconnect; killed in a finally when the turn ends); then ensureSandboxDaemon deploys/verifies the daemon bundle.
Mint the scope-bound token
HMAC-sign { userId, sandboxId, requestId, scope, collectionId?, summaryId?, exp:+10m } with LLM_TOKEN_SECRET (packages/llm-token). One token serves both route families on the gateway.
chat-api → daemon
POST /turn (x-daemon-auth-token) with the system prompt, scope, message, llm_base_url, doc_gateway_url, and llm_token.
Spawn the agent (srt)
Daemon spawns bun agent.js under srt (bwrap on Linux · sandbox-exec on macOS) — filesystem allowlist for the workspace cwd, network allowlist for the gateway hosts only. Env: ANTHROPIC_BASE_URL/ANTHROPIC_AUTH_TOKEN + MYMEMO_DOC_GATEWAY_URL/MYMEMO_DOC_TOKEN, no ANTHROPIC_API_KEY, no DATABASE_URL.
agent → gateway · documents (on demand)
When it needs documents the agent runs mymemo-docs search/fetch (a CLI on its PATH) → the gateway's /v1/documents/* routes verify the token, enforce the signed scope, and read the KB Postgres (FTS over passages) with the read-only DATABASE_URL. Nothing is materialized to the sandbox FS.
claude → gateway · messages
The binary calls ${ANTHROPIC_BASE_URL}/v1/messages with Authorization: Bearer. The gateway's catch-all proxy verifies (signature + expiry + shape), checks the path allowlist, injects x-api-key, and proxies to Anthropic.
Stream back
Anthropic's SSE streams through the gateway untouched → agent emits NDJSON (text_delta, session_id) → daemon forwards → chat-api relays as SSE to the client.

05

Inside the sandbox

The daemon spawns a fresh, isolated process per turn — the agent is a grandchild of the daemon.

daemon.js # long-running Hono server · port 8080 · holds no secrets └─ spawn srt … # per-turn: filesystem allowlist + network allowlist + ns └─ bun agent.js # env: ANTHROPIC_* + MYMEMO_DOC_* (tokens only, no keys) └─ claude # Agent SDK · Authorization: Bearer ├─ Bash / Grep / Read # tool subprocesses ├─ mymemo-docs # CLI on PATH · search/fetch via gateway /v1/documents/* ├─ ✓ gateway /v1/messages # model calls — allowed outbound ├─ ✓ gateway /v1/documents # document calls — allowed outbound (same host) └─ ✗ everywhere else # anthropic, KB Postgres, public internet — denied
06

Runtime variants

Same architecture, three deployment shapes — only the sandbox runtime and the gateway's locality change.

EnvironmentSandbox runtimegateway (llm + doc, one service)Agent isolation
production real E2B · Firecracker microVM one service · public HTTPS srt (bwrap) + egress allowlist
E2B integration test real E2B sibling process inside the sandbox · loopback srt (bwrap) + egress allowlist
local dev · agent + gateway none — bun on host bun on host · localhost (skipped — no sandbox runtime)

The same code path runs in all three — the toggle is GATEWAY_PUBLIC_URL (loopback vs. public URL) plus the choice of sandbox provider. The Claude binary and the mymemo-docs CLI both point at it. chat-api always mints the token; the gateway always verifies it.
